# Frequently Asked Questions
---

Questions are organized into [General](#general) (common setup questions for new users), [Non-default processing](#non-default-processing) (for sessions where something went wrong and/or you may need to deviate from default settings), and [Versioning](#versioning) (for checking what version you're currently on and staying up-to-date with the most recent changes).


Note that a lot of these questions have duplicate info already in `metadata_fully_explained.yaml`. Reading through that file is a good place to start.

# General
---

### What (behavior) files do I need?

For all sessions, you need behavior data. This means you need to specify paths for `arduino_text_file_path`, `arduino_timestamps_file_path`, and `maze_configuration_file_path`. Arduino text and timestamps files are automatically generated by running the task. You need to manually create the file for `maze_configuration_file_path` (see question "What is `maze_configuration_file_path` (aka barriers.txt)?" below).

### What other metadata do I need to specify?

You also need to specify general metadata about the session. This is `institution` (always "University of California, San Francisco"), `lab` (always "Berke lab"), `experimenter` (your name), `experiment_description` (for now, always "Hex maze task", but you can change this if you want), and `date` of the experimental session. Specify dates in either MMDDYYYY or YYYYMMDD format. Slashes and dashes are ok (e.g 07-25-2022, 07/25/2022, 2022-07-25, 2022/07/25, 07252022, 20220725 are all fine). If you do not set `institution` or `lab`, they will default to "University of California, San Francisco" and "Berke lab".

For subject metadata, specify `subject_id` (e.g. IM-1478), `species` (Rattus norvegicus), `sex` (M or F), `genotype` (e.g. Wildtype), `date_of_birth` (same format as date of experiment: MMDDYYYY or YYYYMMDD, slashes and dashes are ok), and `description` (e.g. Long Evans Rat). You can add extra info to the description if you want (such as weight at session start, etc.). If you don't know the date of birth, use the `age` field instead, and specify your best guess for the subject's age in months (e.g. 6mo). 

The extra `animal_name` field is optional, and defaults to `subject_id` if not specified. The nwb file will automatically be named based on the animal name and date (e.g. `IM-1891_jimmy_20250723.nwb` for the example below, because the animal_name is `IM-1891_jimmy`). We generally only use the animal name field if we want the nwb file name to be different than just `subjectid_date.nwb`. If `animal_name` was not specified here, the only difference would be that the nwb in the example below would be named `IM-1891_20250723.nwb` (because the subject_id is `IM-1891`). Adding a suffix to the animal name field is currently how we specify multiple epochs per day (see question "I recorded multiple sessions on the same day. What do I do?").

You can just copy one of the example metadata files for this part, and change things to match your subject and session. It should look like this (copied from `metadata_example_Jose.yaml`): 

```
institution: University of California, San Francisco
lab: Berke Lab
experimenter:
  - Figueroa, Jose
experiment_description: Hex maze task
animal_name: IM-1891_jimmy
date: "20250723"
subject:
  subject_id: IM-1891
  species: Rattus norvegicus
  genotype: Wildtype
  sex: M
  date_of_birth: 2024-10-17
  description: Long Evans Rat
```

### What is `maze_configuration_file_path` (aka barriers.txt)?

You need to manually create a maze configuration file (barriers.txt) for each session. This is just a regular text file. The file should include 1 line per block in that session, expressed as a list of hexes where barriers are placed. (e.g. 8, 9, 16, 18, 27, 23, 35, 42, 44). For probability change sessions, this can be a single line that will be used for all blocks. Any lines starting with `#` in the file will be ignored when the file is parsed, so feel free to add comments as well. For example, my barriers.txt file might look like:

```
# Barrier change session with 3 blocks
# Block 1
35, 11, 12, 45, 14, 15, 18, 22, 29, 31
# Block 2
11, 12, 45, 14, 15, 18, 20, 22, 29, 31
# Block 3
36, 11, 12, 45, 14, 15, 20, 29, 31
```

Or for a probability change session, it might just be one line:

```
10, 9, 18, 19, 25, 37, 36, 34, 45
```

The file can be named whatever you want as long as the `maze_configuration_file_path` path points to it (you don't have to name it barriers.txt if you don't want to). You may choose to write a script that auto-generates this file when you pick your maze configurations, or just type it manually every time you run a session.

### How can I get hex centroids?

See [resources/Get_Hex_Centroids.ipynb](resources/Get_Hex_Centroids.ipynb) to generate a hex centroids file from the session's behavior video. If you don't move the maze in between sessions, you can use the same centroids file for multiple sessions. If you are reusing the same centroids file for multiple sessions, skip to "Part 4" of the centroids notebook to confirm they still line up correctly with your behavior video for this session. The conversion pipeline also produces a sanity check figure of the hex centroids overlaid on the maze video, so you can also use that to confirm that everything matches.

NOTE: If you cropped the video for DeepLabCut position tracking (as we did in some early sessions), use the cropped video to get the centroids, or shift all of the centroids by the crop amount to ensure they match the cropped video.

### How can I run batch conversion of multiple sessions at once?

Here is a script to do this:

```
#!/usr/bin/env bash

# Directory where your metadata files live
METADATA_DIR="/Users/steph/berkelab/jdb_to_nwb/metadata"

# Output directory for created nwbs
OUT_DIR="/Users/steph/berkelab/jdb_to_nwb/output"

# List metadata files you want to convert
FILES=(
    "$METADATA_DIR/1478/metadata_IM-1478_20220725.yaml"
    "$METADATA_DIR/1478/metadata_IM-1478_20220727.yaml"
    "$METADATA_DIR/1594/metadata_IM-1594_20230726.yaml"
)

# Loop through them and convert one by one
for METADATA in "${FILES[@]}"; do
    echo "Starting conversion for: $METADATA"
    jdb_to_nwb "$METADATA" "$OUT_DIR"
    echo "Finished conversion for: $METADATA"
    echo "---------------------------------------"
done

echo "All conversions completed!"
```

1. Save this as `batch_conversion.sh`
2. Make it executable: `chmod +x batch_conversion.sh`
3. Run it: `./batch_conversion.sh`

# Non-default processing

So I did something weird this session, and/or something went wrong. How can I deal with that?

### I recorded multiple sessions on the same day. What do I do?

For now, we handle this using the `animal_name` field described above. Instead of the regular animal name, add a suffix "_2" (or "_3", "_4", etc. for further epochs). You'll end up with file names like `IM-1875_20250725.nwb` (for the first epoch of the day, which you specify normally), `IM-1875_2_20250725.nwb` (for the second epoch, where you set `animal_name: IM-1875_2`), `IM-1875_3_20250725.nwb`..., etc. 


### I stopped one of the recording streams (ephys, behavior, or photometry) before one of the others, so they recorded different numbers of port visits. What do I do?

We already handle this automatically, so it should be fine. You'll see a note in the warning logs. Just double check the debug logs to make sure everything looks fine.

### My photometry recording dropped, so I need to crop it to valid times. What do I do?

Use the notebook [resources/photometry/crop_photometry.ipynb](resources/photometry/crop_photometry.ipynb) to find the desired end time of your photometry signal. Then use the optional metadata field `phot_end_time_mins` to specify the desired end time of your photometry recording (see `metadata_fully_explained.yaml`).

### I moved the barriers at a different time than the arduino printout. What do I do?

We do this a lot. Use optional metadata field `barrier_shift_trial_counts` to specify the trials where you actually moved the barriers (see `metadata_fully_explained.yaml`). The conversion pipeline automatically generates figures showing the rat position on each trial - use these to double-check that the barrier shift trials you specified match the rat's trajectories on the trials near a barrier shift.

### I plugged in the ephys cables the wrong way (when using custom Berke lab probes). What do I do?

Use optional ephys metadata field `plug_order`.

### I want to change the impedance criteria for when an electrode is marked "bad channel". What do I do?

Use optional ephys metadata fields `min_impedance_ohms` (Defaults to 1e5. Channels below this impedance will be tagged as "bad channel") and/or `max_impedance_ohms` (Defaults to 3e6. Channels above this impedance will be tagged as "bad channel").

# Versioning

`jdb_to_nwb` is installed as an editable install (`pip install -e .`), and its
version number is set automatically from the **latest git tag** in your local
history. Keeping the version number up-to-date is really helpful for debugging and for us to know exactly what code and processing steps generated each nwb file.

### What version should I be using?

You should always be using the latest version. **The current latest version is ![latest tag](https://img.shields.io/github/v/tag/calderast/jdb_to_nwb)**.

You can also check the version of the latest release on the [Releases page](https://github.com/calderast/jdb_to_nwb/releases), which will also include details about what's changed since the last version. The latest release number should match the latest tag on the [Tags page](https://github.com/calderast/jdb_to_nwb/tags). If these don't match, the most "up to date" version is the one on the tags page. (I will do my best to create a new release for every new tag. Sometimes I create a new tag without a new release if it's a very minor change. Feel free to ask or bother me if these are not in sync).

Note that commits for small changes will not come with a new tag. This means the version you see may be a dev version: the tag number+1 with `.devN` and some extra info tacked on, telling you you're N commits past the latest tag (e.g. `2.3.2.dev1`, meaning 1 commit past version 2.3.1). This is normal. See "How can I check what version I am using?" and "What does my version number mean?" below for how to read it.

### What does my version number mean?

Your version number is generated automatically from your git history. It comes in one of two shapes.

**A clean version** looks like a plain three-part number with nothing extra:

```
2.3.1
```

This means your local code sits exactly on a tagged commit. You'll see this if
you installed a specific release, or if you pulled and reinstalled right when a
tag was made. If your version looks like this, and the number matches the latest tag, you're perfect and up-to-date.

**A dev version** looks longer, like this:

```
2.3.2.dev1+gefebc8e8c.d20260522
│     │     │          │
│     │     │          └─ "dirty" flag: uncommitted changes were present,
│     │     │             as of this date (2026-05-22)
│     │     └──────────── the git commit you're on (efebc8e8c),
│     │                   the leading "g" just means "git"
│     └────────────────── 1 commit past the most recent tag
└──────────────────────── the next version number (bumped up from the latest tag)
```

| Piece | Meaning |
|---|---|
| `2.3.2` | The next version number, guessed by bumping the patch number up from your latest tag — so the tag you're actually counting from here is `2.3.1` |
| `.dev1` | You are 1 commit past that tag (`2.3.1`) |
| `+gefebc8e8c` | Everything after the `+` is "local" info. `g` just means git, and `efebc8e8c` is the short hash of the commit you're on. |
| `.d20260522` | The `.d<date>` means your working tree was "dirty" (aka you had uncommitted changes) when the package was installed, on 2026-05-22. Make sure you actually want these!! |

**A dev version is not a problem!** If the base number matches (or is one bump above) the latest tag, you're probably up-to-date. You can double check the commit hash (e.g. `efebc8e8c`) matches the hash of the [latest commit on github](https://github.com/calderast/jdb_to_nwb/commits/main/)

### I'm not up to date - how can I fetch the latest changes?

Because the version number gets updated when you install, pulling new code alone won't update your version number. You always need to reinstall afterward (run `pip install -e .`). Every path below ends with that step.

**Before you pull:** if your version string had a `.d<date>` dirty flag, you have
uncommitted changes. Check what they are first using `git status` so you don't lose anything or hit a
merge conflict.

If there's nothing you want to keep, you can use `git stash` to set those changes aside (this removes them from your working directory but saves them. You can then use `git stash pop` later to get them back). You can also just ask me :)

Then follow the case that matches how you installed:

**1. You cloned this repo directly** (the clone URL had `calderast` in it).
You can just pull the latest changes and reinstall:

```bash
git pull
pip install -e .
```

**2. You installed from your own fork.**
First go to your fork on GitHub and click **"Sync fork" --> "Update branch"** to bring your
fork up to date with mine. Then locally pull the latest changes and reinstall:

```bash
git checkout main
git pull
pip install -e .
```

**3. You did #2 but your version is still out of date.**
This is because GitHub's **"Sync fork"** button updates commits but not tags, and the version comes from tags. Point git at this repo (we'll call it "upstream") and grab the tags:

```bash
# this first command is only needed once — skip if you already have an upstream remote
git remote add upstream https://github.com/calderast/jdb_to_nwb.git

git fetch upstream --tags
git pull upstream main --tags
pip install -e .
```

**Confirm it worked:**

```bash
pip show jdb-to-nwb
```

The version should now match the latest tag on the
[Tags page](https://github.com/calderast/jdb_to_nwb/tags). If it's still wrong, lmk and we'll debug it together so I can update this.
